## 1. 프로젝트 개요

### 1.1 프로젝트 목적

개발자 지망생 및 학생들이 기술 스택 기반으로 최적의 팀원을 찾고, 모집부터 승인까지의 전 과정을 자동화하여 효율적인 팀 빌딩을 지원한다.

### 1.2 프로젝트 범위

- **회원 관리**: 가입, 로그인(JWT), 기술 스택 기반 프로필 설정, 마이페이지(활동 이력)
- **모집글 관리**: 프로젝트/스터디 모집글 등록·수정·삭제, 기술 스택별 검색 및 페이징
- **매칭 관리**: 참여 신청서 작성, 방장의 승인/거절 처리, 모집 인원 실시간 카운팅
- **관리자 페이지**: Thymeleaf 기반 전체 게시글 모니터링, 악성 유저 정지 및 데이터 관리

### 1.3 주요 이해관계자

| 구분 | 역할 | 주요 관심사 |
| --- | --- | --- |
| **방장 (Host)** | 프로젝트 개설자 | 적합한 기술 스택을 가진 팀원 선발, 효율적인 모집 관리 |
| **팀원 (Member)** | 프로젝트 지원자 | 본인 스택에 맞는 프로젝트 탐색, 간편한 지원 및 결과 확인 |
| **관리자 (Admin)** | 시스템 운영자 | 부적절한 게시글 및 유저 제재, 서비스 활성 지표 모니터링 |

### 1.4 도메인 설명

 본 시스템은 **팀원 매칭 플랫폼**으로, 사용자가 프로젝트/스터디를 생성하고 필요한 팀원을 모집하여 협업할 수 있도록 지원하는 서비스이다.

사용자는 회원가입 및 로그인을 통해 서비스에 접근하며, 인증은 **JWT 기반 인증 방식**을 사용하여 보안성을 확보한다. 사용자 권한은 관리자(ADMIN)와 일반 사용자(USER)로 구분되며, 프로젝트 내부에서는 추가적으로 방장(OWNER)과 팀원(MEMBER) 역할로 세분화된다. 

일반 사용자는 프로젝트를 생성할 수 있으며, 프로젝트를 생성한 사용자는 해당 프로젝트의 방장(OWNER)이 된다. 방장은 프로젝트에 대한 전반적인 관리 권한을 가지며, 모집글 작성 및 수정, 삭제, 지원자 승인 및 거절, 팀원 초대 및 관리 등의 기능을 수행할 수 있다.

다른 사용자들은 프로젝트 목록을 조회하고 관심 있는 프로젝트에 지원할 수 있다. 지원은 Application 도메인으로 관리되며, 지원 상태는 대기(PENDING), 승인(APPROVED), 거절(REJECTED)로 구분된다. 방장이 지원을 승인하면 해당 사용자는 프로젝트의 팀원(MEMBER)으로 등록된다.

프로젝트는 최대 인원 수 제한을 가지며, 정원이 초과되지 않도록 관리된다. 팀원은 프로젝트 참여 상태(Membership)를 가지며, 자발적으로 탈퇴할 수 있고, 방장에 의해 강제 퇴출될 수도 있다.

프로젝트 내부에는 게시글(Post) 기능이 존재하여 팀원 간 소통이 가능하며, 게시글에는 댓글(Comment)을 작성할 수 있다. 게시글 및 댓글은 작성자 또는 방장에 의해 수정 및 삭제가 가능하며, 관리자는 모든 게시글에 대해 삭제 권한을 가진다.

또한, 사용자는 프로필 이미지와 같은 파일을 업로드할 수 있으며, 이를 통해 개인 정보를 관리할 수 있다. 시스템은 예외 처리 및 에러 코드를 일관되게 관리하여 안정적인 서비스 제공을 목표로 한다.

이와 같은 도메인 구조를 통해 사용자 간 협업을 효율적으로 지원하며, 권한 기반 접근 제어와 상태 관리를 통해 실제 서비스와 유사한 비즈니스 로직을 구현한다.

## 2. 비즈니스 도메인 분석

### 2.1 핵심 비즈니스 프로세스

### 2.1.1 회원 관리 및 인증 프로세스

**상세 플로우:**

1. **회원가입 단계**
    - 기본 정보 입력 (이메일, 닉네임, 비밀번호, 전화번호)
    - 초기 온보딩: 선호 직군(FE/BE 등) 및 보유 기술 스택(Java, React, 신입 등) 선택
    - 비밀번호 BCrypt 암호화 및 유저 엔티티 생성
    - 기본 프로필 이미지 자동 할당
2. **인증 및 활동 단계**
    - 로그인 시 JWT(Access/Refresh Token) 발급
    - 마이페이지를 통한 기술 스택 및 포트폴리오 링크 수정
    - 나의 활동 이력 조회 (내가 작성한 모집글, 내가 지원한 신청 내역 상태 확인)
3. **계정 찾기 단계**
    - **아이디 찾기:** 전화번호를 입력받아 DB의 User 엔티티와 대조 후 일치하는 이메일 반환
    - **비밀번호 찾기:** 이메일과 전화번호의 일치 여부 검증 ➔ 8자리 이상의 랜덤 임시 비밀번호 생성 ➔ BCrypt 암호화 후 User 엔티티의 비밀번호 업데이트 ➔ 사용자에게 평문 임시 비밀번호 제공

### 2.1.2 팀 빌딩 및 매칭 프로세스

**상세 플로우:**

1. **모집 공고**: 방장(Host)이 프로젝트 상세 정보와 필요한 기술 스택, 모집 인원을 설정하여 게시글 등록
2. **지원 신청**: 팀원(Member)이 게시글 상세 페이지에서 지원 동기를 입력하여 신청 (`PENDING` 상태)
3. **매칭 검토**: 방장이 신청자 리스트와 각 신청자의 기술 스택 프로필을 대조/검토
4. **최종 확정**: 방장이 승인(`ACCEPTED`) 시 현재 모집 인원이 증가하며, 정원 충족 시 해당 공고는 자동으로 마감(`CLOSED`) 처리

**비즈니스 규칙:**

- **중복 지원 금지**: 동일한 사용자는 한 게시글에 단 한 번만 지원 가능
- **본인 지원 금지**: 게시글 작성자(방장)는 본인이 올린 공고에 지원할 수 없음
- **지원 제한**: 무분별한 지원 방지를 위해 유저당 `PENDING` 상태의 지원서는 최대 8개로 제한
- **모집 정원**: 최소 1명에서 최대 20명까지 설정 가능 (작성자 본인은 카운트에서 제외)

### 2.2 비즈니스 이벤트

| 이벤트 | 트리거 | 결과 |
| --- | --- | --- |
| **회원 가입 완료** | 가입 폼 제출 성공 | 유저 엔티티 생성, ROLE_USER 권한 부여, 환영 메시지 노출 |
| **모집글 게시** | 작성 완료 버튼 클릭 | Post 엔티티 생성 (RECRUITING 상태), 기술 스택 태그 매핑 |
| **참여 신청 발생** | 지원서 제출 | Application 엔티티 생성 (status=PENDING), 방장에게 알림 표시 |
| **매칭 승인** | 방장의 승인 버튼 클릭 | Application status: ACCEPTED, Post currentCount 증가 |
| **매칭 거절** | 방장의 거절 버튼 클릭 | Application status: REJECTED, 지원자에게 결과 통보 |
| **모집 자동 마감** | currentCount == recruitCount | Post status: CLOSED 변경, 추가 지원 버튼 비활성화 |
| **강제 게시글 삭제** | 관리자 또는 방장의 삭제 | Post status: DELETED (Soft Delete), 관련 신청 내역 무효화 |

## 3. 상세 도메인 객체 정의 (DB 설계 및 Data Schema)

### 3.1 Users (사용자)

- 서비스 이용자의 로그인 인증 데이터 및 기술 스택 프로필을 관리 (email 유니크 인덱스 설정)

| 속성명 | 타입 | 설명 | 제약 조건 |
| --- | --- | --- | --- |
| `email` | String | 로그인 ID = 서비스 활동명 | 중복 불가, 이메일 형식 |
| `nickname` | String | 서비스 활동명 | 2~10자, 중복 불가 |
| `password` | String | 암호화된 비밀번호 | BCrypt 암호화 필수 |
| `position` | Enum | FE, BE, DE, PM, ETC | 선택 필수 |
| `techStacks` | List | 보유 기술 스택 태그 (신입 권한도 생성) | `user_tech_stacks` 별도 테이블로 1:N 분리 매핑 |
| `profileImg` | String | 프로필 이미지 경로 | 기본 이미지 제공 |
| `phone_number` | String | 휴대폰 번호  | 중복 불가, 하이픈 포함/미포함 형식 검증 |

### 3.2 Projects (프로젝트/모집글)

- 방장이 생성하는 프로젝트 본체로, 모집 유형, 목표 인원(`recruitCount`) 대비 현재 인원(`currentCount`) 및 모집 상태를 관리

| 속성명 | 타입 | 설명 | 제약 조건 |
| --- | --- | --- | --- |
| `projectId` | Long | 프로젝트 식별자 | PK |
| `category` | Enum | STUDY, PROJECT | 선택 필수 |
| `title` | String | 모집 제목 | 5~50자 |
| `content` | Text | 모집 상세 내용 | - |
| `recruitCount` | Integer | 목표 모집 인원 | 1~20명 |
| `currentCount` | Integer | 현재 확정 인원 | 기본값 0, 작성자 미포함 |
| `onOffline` | Enum | ONLINE, OFFLINE, BOTH | 활동 방식 |
| `status` | Enum | RECRUITING, CLOSED, DELETED | 기본값 RECRUITING |
| `endDate` | LocalDate | 모집 마감일 | 과거 날짜 선택 불가 |
| `ownerId` | Long | 방장(작성자) ID | FK (User) |

### 3.3 Applications (지원서)

- 유저와 프로젝트 간의 다대다(N:M) 관계를 해소하고, 매칭 상태(대기/승인/거절) 이력을 관리

| 속성명 | 타입 | 설명 | 제약 조건 |
| --- | --- | --- | --- |
| `applicationId` | Long | 지원서 식별자 | PK |
| `projectId` | Long | 지원한 프로젝트 ID | FK (Project) |
| `applicantId` | Long | 지원자 ID | FK (User) |
| `message` | String | 지원 동기 및 자기소개 | 10~500자 |
| `status` | Enum | PENDING, ACCEPTED, REJECTED | 기본값 PENDING |
| `appliedAt` | DateTime | 지원 일시 | 자동 생성 |

### 3.4 Project_Members (팀 멤버 / 멤버십)

- 방장의 승인으로 프로젝트에 최종 합류한 인원을 관리하며, 방장(OWNER)과 팀원(MEMBER)의 권한을 구분하여 게시판 접근 제어의 근거로 활용

| 속성명 | 타입 | 설명 | 제약 조건 |
| --- | --- | --- | --- |
| `memberId` | Long | 멤버십 식별자 | PK |
| `projectId` | Long | 한 프로젝트 ID | FK (Project) |
| `userId` | Long | 사용자 ID | FK (User) |
| `role` | Enum | OWNER, MEMBER | 방장/일반팀원 구분 |
| `joinedAt` | DateTime | 참여(승인) 일시 | - |

### 3.5 Board_Posts (프로젝트 내부 게시글)

- 매칭이 완료된 팀원 전용 내부 게시판
- 공지 및 자료 공유를 지원하며 Soft Delete를 통해 데이터 이력을 보존

| 속성명 | 타입 | 설명 | 제약 조건 |
| --- | --- | --- | --- |
| `postId` | Long | 게시글 식별자 | PK |
| `projectId` | Long | 소속 프로젝트 ID | FK (Project) |
| `authorId` | Long | 작성자 ID | FK (User) |
| `title` | String | 게시글 제목 | 1~100자 |
| `content` | Text | 게시글 내용 (마크다운 등) | 필수 입력 |
| `createdAt` | DateTime | 작성 일시 | 자동 생성 |
| `updatedAt` | DateTime | 수정 일시 | - |
| `deletedAt` | DateTime | 삭제 일시 | Soft Delete 처리용 |
| `viewCount` | `Integer` | 조회수 (인기글 정렬용) | Default 0 |

### 3.6 Comments (게시글 댓글)

- 내부 게시글에 대한 팀원들의 유기적인 피드백과 소통을 위한 댓글 시스템

| 속성명 | 타입 | 설명 | 제약 조건 |
| --- | --- | --- | --- |
| `commentId` | Long | 댓글 식별자 | PK |
| `postId` | Long | 소속 게시글 ID | FK (BoardPost) |
| `authorId` | Long | 작성자 ID | FK (User) |
| `content` | String | 댓글 내용  | 1~500자, 필수 입력 |
| `createdAt` | DateTime | 작성 일시 | 자동 생성 |
| `updatedAt` | DateTime | 수정 일시 | - |
| `deletedAt` | DateTime | 삭제 일시 | Soft Delete 처리용 |

## 4. 도메인 관계도 (ERD)

![Gemini_Generated_Image_ikd2brikd2brikd2.png](Gemini_Generated_Image_ikd2brikd2brikd2.png)

### **4.1 USER 와 PROJECT (생성 및 관리 권한 관계):** `1:N`

- **논리적 의미:** 한 명의 회원(USER)은 방장으로서 여러 개의 프로젝트(PROJECT)를 개설할 수 있다. 반면, 하나의 프로젝트는 단 한 명의 개설자(OWNER)만 가진다.
- **비즈니스 역할:** `PROJECT` 테이블은 방장의 식별자인 `owner_id`를 외래키(FK)로 가진다. 이 관계를 통해 시스템은 **"누가 이 프로젝트의 지원자를 수락/거절할 수 있는지"**, "누가 모집글을 수정/삭제/마감할 수 있는지"에 대한 절대적인 권한(Authority)을 판별한다.

### **4.2 USER 와 PROJECT (매칭/지원 관계):** `N:M` ➔ `APPLICATION` 테이블로 해소

- **논리적 의미:** 한 명의 회원은 여러 프로젝트에 지원할 수 있고, 하나의 프로젝트에는 여러 회원이 지원할 수 있는 다대다(N:M) 관계이다.
- **비즈니스 역할:** 이를 해결하기 위해 중간에 `APPLICATION` (지원서) 테이블을 두어 `1:N`, `N:1` 관계로 풀어냈다. 이 테이블은 단순히 연결만 하는 것이 아니라, 지원자의 자기소개(message)와 **현재 매칭 상태(PENDING, ACCEPTED, REJECTED)**, 그리고 **지원 일시**라는 핵심 비즈니스 데이터를 품고 있는 서비스의 가장 중요한 매칭 이력 저장소다.

### **4.3 USER 와 PROJECT (멤버십/소속 관계):** `N:M` ➔ `PROJECT_MEMBER` 테이블로 해소

- **논리적 의미:** 지원서(APPLICATION)가 방장에 의해 승인(ACCEPTED)된 후, 최종적으로 프로젝트에 합류한 '진짜 팀원'들을 관리하는 관계이다.
- **비즈니스 역할:** `PROJECT_MEMBER` 테이블은 승인된 유저들만 기록되는 **'프로젝트 전용 출입증'** 역할을 한다. 멤버십의 권한(`role`: OWNER / MEMBER)을 저장하며, 추후 이 테이블에 등록된 유저만이 해당 프로젝트의 내부 소통 게시판(Board_Post)에 접근하고 글을 쓸 수 있도록 철저한 접근 제어(Access Control)의 기준이 된다. 동일 유저가 중복 가입되지 않도록 복합 유니크 키(project_id + user_id)가 적용된다.

### 4.4 USER 와 USER_TECH_STACK (기술 스택 관계) : `1:N`

- **논리적 의미:** 한 명의 회원은 Java, React, Spring 등 여러 개의 기술 스택을 보유할 수 있다.
- **비즈니스 역할:** 관계형 데이터베이스의 정규화 원칙(제1 정규형)을 준수하기 위해 기술 스택을 콤마(,)로 이어 붙인 문자열로 저장하지 않고, 별도의 `USER_TECH_STACK` 테이블로 분리했다. 이를 통해 추후 프론트엔드에서 "React를 할 줄 아는 사람"이나 "Spring 스터디"를 검색할 때 빠르고 정확한 필터링 및 조인(Join) 쿼리가 가능해진다.

### 4.5 PROJECT 와 BOARD_POST / COMMENT (내부 소통 관계) : `1:N`

- **논리적 의미:** 하나의 프로젝트(PROJECT) 내부에 여러 개의 게시글(BOARD_POST)이 작성될 수 있으며, 하나의 게시글에는 여러 개의 댓글(COMMENT)이 달릴 수 있다.
- **비즈니스 역할:** 매칭이 완료된 후 팀원들끼리 자료를 공유하고 피드백을 주고받는 협업 공간이다. 모든 게시글과 댓글은 작성자(`author_id` ➔ USER)와 소속 프로젝트(`project_id` ➔ PROJECT)를 동시에 참조하여 데이터의 출처를 명확히 한다.

## 5. 상태(State) 생명주기 정의

### 5.1 Project (프로젝트) 상태

- **RECRUITING (모집 중):** 생성 직후 상태. 사용자들이 지원할 수 있다
- **CLOSED (모집 마감):** `currentCount`가 `recruitCount`에 도달하거나 방장이 수동으로 마감한 상태. 추가 지원 불가
- **DELETED (삭제됨):** Soft Delete 처리되어 목록에서 노출되지 않는 상태

### 5.2 Application (지원서) 상태

- **PENDING (대기 중):** 지원서를 제출하고 방장의 응답을 기다리는 상태
- **ACCEPTED (승인됨):** 방장이 수락하여 `ProjectMember`에 데이터가 생성된 상태
- **REJECTED (거절됨):** 방장이 거절했거나, 프로젝트가 CLOSED 되어 자동으로 처리된 상태

## 6. 상세 비즈니스 규칙 (Business Rules)

### 6.1 모집글 관련 (Post Rules)

- **PR-01**: 작성자(방장)는 본인의 글을 수정할 수 있으나, 이미 승인된 인원이 있을 경우 '모집 인원'을 현재 인원보다 적게 수정할 수 없다.
- **PR-02**: 게시글 삭제 시 관련 모든 신청서(`Application`)는 함께 삭제되거나 취소 상태로 변경되어야 한다 (Soft Delete 권장)
- **PR-03**: 마감된 게시글(`CLOSED`)은 목록 상단에서 제외되거나 별도의 표시를 가진다.
- **PR-04 (기간 만료 마감):** 설정된 모집 마감일(`endDate`) 자정이 지나면, 정원 충족 여부와 관계없이 해당 모집글의 상태는 `CLOSED`로 자동 변경되며 더 이상 지원을 받지 않는다.

### 6.2 매칭 및 지원 관련 (Matching Rules)

- **MR-01**: 본인이 작성한 게시글에는 지원 버튼이 비활성화되거나 노출되지 않는다.
- **MR-02**: 한 유저는 동시에 최대 **8개**의 게시글에 `PENDING`(대기) 상태로 지원 가능하다. (무분별한 지원 방지 및 기회 확대)
- **MR-03**: 방장이 모집을 마감하거나 정원이 충족될 경우, 해당 글의 모든 `PENDING` 신청은 시스템에 의해 자동으로 `CLOSED` 상태로 전환된다.
- **MR-04**: 지원자(Member)는 본인이 신청한 내역이 `PENDING`(대기) 상태일 때만 지원을 취소할 수 있다.
- **MR-05**: 자동 취소된 신청 건은 사용자의 '마이페이지 > 지원 내역'에서 상태 변화를 확인할 수 있도록 하며, 필요 시 알림 창을 통해 안내한다.
- **MR-06 (재지원 원천 차단):** 방장에게 한 번 **'거절(REJECTED)'** 당했거나, 프로젝트에서 **'강제 퇴출(KICKED)'** 처리된 유저는 해당 모집글에 다시 지원할 수 없다.

### 6.3 권한 및 기타 (Authority)

- **AR-01**: 게시글 수정/삭제 및 지원자 승인/거절은 해당 글을 작성한 '방장'만 가능하다.
- **AR-02**: 기술 스택을 최소 1개 이상 등록하지 않은 유저는 프로젝트에 지원할 수 없다.
- **AR-03**: 관리자(ADMIN)는 부적절한 게시글을 강제로 마감하거나 삭제할 수 있는 권한을 가진다.

### 6.4 팀원 관리 및 소통 규칙 (Member & Board Rules)

- **MMR-01 (멤버 강퇴):** 방장(OWNER)은 악의적인 팀원(MEMBER)을 프로젝트에서 강제 퇴출시킬 수 있는 권한을 가진다.  퇴출 시 `ProjectMember`에서 삭제되고 `currentCount`가 1 감소한다.
- **MMR-02 (방장 위임/탈퇴 불가):** 방장은 본인이 개설한 프로젝트를 임의로 탈퇴할 수 없다.  탈퇴를 원할 경우 프로젝트 자체를 삭제하거나 다른 팀원에게 방장 권한을 위임해야 한다.
- **BDR-01 (게시판 접근 권한):** 프로젝트 내부 게시글(BoardPost)과 댓글(Comment)은 해당 프로젝트의 `ProjectMember`에 등록된 인원만 조회 및 작성할 수 있다.
- **BDR-02 (수정/삭제 권한):** 게시글과 댓글은 작성자 본인만 수정 가능. 그리고 본인, 방장, 관리자(ADMIN)만 삭제할 수 있다.

## 8. 용어 정리 (Glossary)

- **Host (방장)**: 모집글을 생성하고 팀원을 선발하는 권한을 가진 사용자
- **Applicant (지원자)**: 특정 모집글에 참여 의사를 밝힌 유저
- **Tech Tag (기술 태그)**: React, Spring, Python 등 매칭의 핵심 기준이 되는 키워드
- **Soft Delete**: DB에서 데이터를 실제 삭제하지 않고 `is_deleted` 플래그만 변경하는 방식
